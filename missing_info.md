## 第 2 章 · 产品功能定义

**已明确 — 核心功能列表：**

| 功能 | 说明 | 定位 |
|---|---|---|
| AI 智能行程生成 | 输入目的地、日期、偏好和攻略截图，AI 自动生成完整的多日行程（含地点、时间、交通、三餐推荐和预算） | 核心 |
| 天气驱动的实时重规划 | 每 3 分钟自动获取当地天气，当天气恶化（转雨、高温、大风）时自动提醒并支持一键 AI 调整行程 | 核心 |
| 交互式地图 | 在地图上展示每日路线、地点标记、交通方式；支持按天切换视图 | 核心 |
| 地点搜索与探索 | 在地图上自由点击查看周边兴趣点，支持关键词搜索地点详情 | 核心 |
| 攻略截图识别 | 上传小红书、携程等平台的攻略截图，AI 从中提取地点和偏好信息 | 核心 |
| 手动修改行程 | 支持替换或删除行程中的任意地点，自动重新计算路线和通勤时间 | 辅助 |
| 天气图层叠加 | 在地图上叠加降水云图和云层图，直观了解天气分布 | 辅助 |
| 气象预警通知 | 天气变化时发送系统通知，提醒用户考虑调整行程 | 辅助 |
| 旅行海报生成 | 为行程中的每个地点自动拉取实拍照片，生成旅行海报 | 辅助 |
| 行程筛选与搜索 | 按城市、天数筛选已保存的行程，支持关键词搜索 | 辅助 |
| 共同编辑邀请 | 生成包含完整行程信息的邀请文本，复制给同行人 | 辅助 |

**已明确 — AI 实现细节：**
- **模型**：Google Gemini 2.5 Flash，通过 REST API 调用
- **生成策略**：温度参数设为 0.35（保证输出稳定和结构化），强制要求返回 JSON 格式
- **多模态**：支持同时输入文本提示词和攻略截图的 base64 编码，让 AI 从截图中提取地点信息
- **提示词设计**：约 270 行的详细提示词，包含 9 条规划原则（空间顺路、真实坐标、交通方式需具体到线路站名等），要求为每个地点标注室内/雨天友好/天气敏感属性，方便后续雨天重规划
- **输出校验**：服务端对 AI 返回结果执行多层校验——验证地点数量不为空、日期范围合理、地点 ID 统一规范、时间线引用正确，非法字段自动填入安全默认值
- **联网搜索**：AI 本身不联网，但 AI 生成结果中不准确的坐标和营业时间可由用户手动用 Google 地图搜索真实信息替换

**待补充：**
- 截图识别功能的前端上传链路（目前图片选择后尚未完成到 base64 的转换和传给后端的流程）
- 是否考虑切换到国产大模型（如通义千问、DeepSeek）以提升国内访问稳定性

## 第 3 章 · AI 介入与控制

**已明确 — 天气监控机制：**
- 进入行程详情页后，每 **3 分钟** 自动获取一次天气数据
- 天气严重程度分为 **4 级**：正常 → 关注 → 较严重 → 严重
- 触发 AI 提醒的阈值：严重程度从良好跳变到需关注、或连升两级以上、或温度骤变超过 8°C、或天气类型改变且新天气等级较高
- 天气转好时也会提示，建议恢复户外景点

**已明确 — AI 工作状态文案：**

| 场景 | 文案 |
|---|---|
| 首次生成行程中 | "正在生成行程..." |
| 生成失败 | "大模型规划失败：[具体原因]" |
| AI 重规划中 | "正在读取天气并调整行程..." |
| 重规划成功 | "已根据当前天气更新行程，地图和下方路线已同步。" |
| 天气恶化提醒 | "天气从「晴」变为「中雨」，建议让 AI 重新调整户外和交通安排。" |
| 天气转好提醒 | "天气从「中雨」转为「多云」，可以考虑恢复户外景点或优化步行路线。" |
| 目的地搜索中 | "正在搜索地图地点..." |

**已明确 — 用户可执行的手动操作：**
- 替换行程中的任意地点（在地图上搜索真实地点替换 AI 生成的）
- 删除不想去的地点
- 忽略天气告警（将当前天气设为新的基准线）
- 通过自然语言向 AI 描述需求（如"下雨了，把下午户外景点换成室内"）

**已明确 — AI 幻觉防御手段：**
- 服务端严格限制输出格式：地点类别只能是 5 种之一，行程天数 1-14 天，人数 1-20 人
- 如果 AI 返回的地点坐标缺失或为 0，系统不会崩溃，而是标记"缺少可用坐标"并继续
- 用户可随时用 Google 地图搜索真实地点来替换 AI 猜测的结果，这是对抗幻觉的核心手段
- 后端连接失败时自动尝试 3 个不同地址；AI 不可用时还有一套完整的杭州本地演示数据兜底

**待补充：**
- 堵车等实时交通状况的监测（目前未实现）
- AI 生成的公交线路名和站名是否真实存在，目前未做二次核验
- 截图 OCR 结果的准确性验证（截图内容是否被正确识别）

## 第 5 章 · 技术实现说明

**已明确 — 技术栈总览：**

| 层面 | 所用技术 |
|---|---|
| 运行平台 | 华为鸿蒙 (HarmonyOS)，面向手机设备 |
| 前端语言与框架 | ArkTS + ArkUI 声明式 UI |
| 后端 | Node.js（使用原生 http 模块，零第三方依赖） |
| AI 模型 | Google Gemini 2.5 Flash |
| 地图服务 | Google Maps（JavaScript API / Places API / Directions API / Static Maps API），通过 WebView 内嵌网页形式加载 |
| 天气数据 | OpenWeatherMap（实时天气接口 + 降水与云层瓦片图层） |
| 系统通知 | 鸿蒙原生通知服务 (NotificationKit) |
| 图片选择 | 鸿蒙系统照片选择器 |
| 剪贴板 | 鸿蒙系统剪贴板服务 |
| 权限 | 网络访问、大致位置、精确定位 |

**已明确 — 架构特色：**

1. **AI + 本地规则双轨制**：远程由 Gemini 生成结构化行程，本地在用户手动修改行程时由前端直接用数学公式计算地点间球面距离并推断交通方式，无需联网
2. **多后端容错**：前端依次尝试 3 个不同网络地址的后端（局域网 / 模拟器 / 本机），服务端调用 Gemini 也有 curl 和 fetch 两套通道互为备份
3. **地图与原生双向通信**：地图在 WebView 中以网页形式运行，通过 JavaScript 桥与 ArkTS 原生代码互通——点击地图上的地点可以直接定位到行程列表中的对应卡片
4. **天气图层叠加**：在 Google 地图上叠加降水云图和云层的半透明瓦片，地图右上角浮窗显示实时天气，随地图拖动自动更新
5. **AI 幻觉多层防御**：服务端对 AI 输出进行严格的结构和内容校验，不合规的字段自动纠正；前端允许用户随时用 Google 地图真实数据替换 AI 生成的任何地点

**待补充：**
- 数据持久化方案（当前行程仅保存在内存中，应用重启后数据丢失，急需接入鸿蒙本地数据库）
- 离线使用方案（无网络时如何保证基本可用性）
- 是否计划支持 iOS/Android 跨平台
- AI API 请求的完整链路图（从前端发起 → 后端转发 → Gemini 返回 → 校验 → 前端渲染）

## 第 7 章 · 失败应对与反思

**已明确 — 已有应对方案：**

1. **大模型返回异常**
   - AI 返回的不是 JSON → 服务端自动从文本中提取 JSON 部分
   - AI 返回的字段不合规 → 自动填入安全默认值（如地点类别不对就标为"景点"）
   - AI 服务完全不可用 → 三个地址依次尝试，全失败则向用户展示明确错误提示

2. **AI 生成的地点信息不准确（幻觉）**
   - 用户在地点卡片上左滑 → 选择"替换" → 在 Google 地图中搜索真实地点 → 一键替换，坐标、地址、营业时间全部更新为官方数据

3. **天气突然恶化**
   - 自动检测 → 系统通知 + AI 助手入口变黑并亮红点 → 用户点击"让 AI 重新规划" → 行程中的户外地点自动替换为室内或雨天友好地点
   - 天气转好时同样提醒，建议恢复户外安排

4. **Google 地图服务异常**
   - API Key 未配置或失效 → 自动降级为 Google 地图网页版跳转链接
   - 地点照片拉取失败 → 改用 Google 静态地图生成的坐标图作为替代
   - 地点搜索无结果 → 返回空列表，不影响其他功能

**待补充 — 当前已知的局限性：**

1. **无本地数据存储**：行程全在内存中，应用退出或重启后用户花时间调整的计划全部丢失。这是当前最影响实际使用的缺陷。

2. **截图识别链路未走通**：界面上的截图上传入口已经做好，图片也能选出来，但从图片到传给 AI 分析这一步的编码和传输流程还没接上。

3. **多人协同仅为占位**：虽然设计了"邀请共同编辑"功能，但实际只是把行程信息复制到剪贴板，没有实现真正的实时同步编辑。

4. **完全依赖外网**：Google 全套服务（Gemini、Maps、Places）加上 OpenWeatherMap 天气服务，在中国大陆网络环境下都需要可靠的代理方案，且任一服务出问题都会影响对应功能。

5. **手绘风格路线图已移除**：早期设计中有一个将地图渲染为手绘风格的功能（铅笔、水彩、水墨、复古四种风格），相关代码在开发过程中已被删掉。

6. **鸿蒙生态限制**：Google Maps 无法使用原生地图 SDK（鸿蒙不支持 Google Play 服务），只能通过 WebView 加载网页版地图，在大数据量标记和复杂路线渲染时性能受限于 WebView。
